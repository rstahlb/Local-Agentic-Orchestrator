Automate retail inventory tracking using UHF RFID tags. Every item tagged at receiving, deactivated at sale, cross-referenced quarterly — producing a verified shrinkage report showing stolen, damaged, and lost goods. Tag Technology · Hardware · Software Architecture · Build Timeline.

🏷️ Why UHF RFID for a Large Store

For any retail environment larger than a small boutique, UHF RFID is the only technology that makes automated inventory tracking economically and operationally viable. It reads at 10–30 feet, requires no line-of-sight, and can scan hundreds of tags simultaneously.

This is the industry standard used by Walmart, Target, H&M, and Zara globally. The read range and bulk scanning capability make it the only viable option for end-of-quarter full-store inventory walks.

How UHF RFID Works

UHF RFID operates in the 860–960 MHz frequency range. Tags are passive — they have no battery and contain only a tiny antenna and chip. When a reader emits a radio signal, any tag within range harvests that energy, powers up for a millisecond, and broadcasts its unique EPC (Electronic Product Code).

01
Item arrives at receiving dock
Blank RFID label attached. Staff presses one button — tag encoded with SKU, vendor, cost, arrival date in under 2 seconds.
02
Item goes to shelf
Tag is completely passive. Costs nothing to maintain.
03
Point of Sale
Cashier scans barcode OR taps RFID reader. System marks tag as SOLD and records transaction.
04
Exit Portal
Fixed readers scan every tag on every person walking out in under 100ms. ACTIVE tags fire an alert immediately.
05
Quarterly Walk
Staff walks every aisle with handheld reader for 20–40 minutes. System compares against database and generates shrinkage report.

Section 04 Software Architecture

LayerDescription
DatabaseSQLite (small store) or PostgreSQL (medium/large). Stores: items, vendors, RFID tag mappings, sale events, damage logs, walk snapshots, shrinkage reports.
API ServerPython Flask or Node.js Express. Receives RFID events from readers (via LLRP protocol), sale events from POS, and commands from the iPad/Windows apps. REST API with JSON.
RFID Reader BridgeSmall daemon process that connects to fixed readers via LLRP (Low-Level Reader Protocol — the industry standard). Translates reader events into API calls.
iPad AppNative Swift/UIKit app. Used by: receiving staff (tag items), floor staff (mark damage, look up items), managers (view alerts, run reports).
Windows AppWeb app (Chrome) or .NET WinForms app. Used at: POS station, receiving desk, manager office. Same API backend as iPad.
Report EnginePython script that runs on the server at quarter-end. Queries the database, calculates shrinkage, generates a PDF report, and emails it.

API Endpoints

POST /tags/encode       → Encode new RFID tag at receiving (sku, vendor_id, cost, epc)
POST /tags/sell         → Mark tag as sold (epc, sale_price, cashier_id)
POST /tags/damage       → Mark tag as damaged (epc, reason, notes, photo_url)
POST /rfid/event        → Raw RFID read from fixed reader bridge (epc, reader_id, timestamp)
GET  /tags/{epc}        → Look up any tag by EPC → item, vendor, status, history
POST /vendors           → Add a new vendor
GET  /vendors/{id}/report → Vendor analytics: items in, sold, missing, loss rate
POST /walks/start       → Begin a quarterly walk session
POST /walks/{id}/tag    → Record a tag found during walk (epc, zone)
POST /walks/{id}/complete → Complete walk and trigger shrinkage calculation
GET  /reports/quarter/{q} → Full quarterly shrinkage report as JSON
POST /reports/quarter/{q}/pdf → Generate and email the quarterly PDF reportREST

Implementation Checklist

  • Define store zones — name every area: aisles, fitting rooms, stockroom, receiving, exit
  • Decide on tag type — paper for cheap items, polyester for expensive, metal-mount for hard goods
  • Purchase hardware — reader, printer, handheld, tags
  • Set up local server — any PC or Mac running 24/7, or a NAS device
  • Run database setup script — creates all tables
  • Start the Flask API server and test: POST /tags/encode with a test tag
  • Build iPad app in Xcode — set server IP in Settings screen — verify connection
  • Pair iPad to Zebra ZD620 via Bluetooth — test print one label
  • Mount exit portal antennas at door height — one each side of exit, angled inward
  • Train receiving staff on iPad receiving workflow — 30 minutes
  • Tag all existing inventory — one-time effort, typically 1–3 days
  • Go live — begin tagging all new items at receiving
